# Dao Architecture

## 1. Architecture Goal

Dao should be built as a long-term maintainable desktop application.

The architecture must support:

- local-first data
- cross-platform desktop app
- React-based UI
- Go-based business logic
- SQLite persistence
- future AI features
- future cloud sync
- official extension system

The first version should stay simple while preserving clear boundaries for future scaling.

## 3. Boundary Rules

### React Renderer

React is responsible for:

- rendering UI
- user interactions
- client-side routing
- local UI state
- API calls
- data validation at boundaries
- command palette
- extension UI registration

React should not contain core business logic.

### Electron Main

Electron is responsible for:

- creating windows
- managing app lifecycle
- managing preload scripts
- starting and stopping Go local service
- native menus
- tray support later
- auto update later
- secure IPC bridge

Electron should remain thin.

Business logic should not be placed in Electron Main.

### Go Local Service

Go is responsible for:

- business logic
- data persistence
- SQLite access
- migrations
- search
- local file operations
- AI workflow execution later
- sync engine later

Go is the core domain layer of Dao.

### SQLite

SQLite is responsible for:

- local persistence
- offline-first usage
- full-text search with FTS5
- future sync metadata storage

## 4. Communication Model

The recommended first-stage communication model:

```txt
React Renderer
  ↓ HTTP
Go Local Service
  ↓ SQL
SQLite
```

Electron starts the Go local service and passes connection information to React through preload.

### Why local HTTP API

Local HTTP is preferred for the first version because:

- it is simple to debug
- it mirrors future cloud API design
- it avoids complex IPC-only architecture
- it lets Go own business logic cleanly
- it makes testing easier

### Go HTTP Routing

Start with the Go standard library `net/http`.

Use internal helper packages for repeated HTTP concerns, such as JSON responses and error responses.

Consider switching to `chi` when the local API grows enough that the standard library starts creating friction:

- route groups become repetitive
- path parameters are needed across multiple modules
- authentication middleware is introduced
- request logging or recovery middleware is needed
- handlers start doing too much manual routing work

Prefer `chi` over heavier frameworks such as Gin, Echo, or Fiber for the local service. Dao's local service should stay close to `net/http` unless the API surface clearly needs routing and middleware help.

## 5. Local Service Startup

Startup flow:

```txt
1. Electron app starts
2. Electron Main finds an available local port
3. Electron Main generates a session token
4. Electron Main starts Go service with port and token
5. Go service opens SQLite database
6. Go service runs pending migrations
7. Go service exposes /health
8. Electron waits for /health
9. Electron opens React renderer
10. React receives API base URL and token through preload
```

## 6. Security Model

The local service should only listen on:

```txt
127.0.0.1
```

Every request should include a local session token:

```txt
Authorization: Bearer <token>
```

The token is generated by Electron at startup and passed to the Go service.

The React renderer receives the token through preload.

### Security rules

- never expose Node APIs directly to React
- use `contextBridge` in preload
- keep `nodeIntegration` disabled
- keep `contextIsolation` enabled
- local service listens only on localhost
- all local HTTP APIs require token authentication

## 8. Data Model Principles

All major entities should include sync-ready fields from the beginning.

Recommended common fields:


### ID Strategy

Use ULID for entity IDs.

Reasons:

- can be generated locally
- sortable by time
- sync-friendly
- easier debugging than random UUID

## 11. JavaScript Type Safety Strategy

Dao uses JavaScript instead of TypeScript.

To keep the code scalable, use:

```txt
JSDoc
Zod
ESLint
runtime validation
clear API contracts
```

### JSDoc

Use JSDoc for shared object shapes.

Example:

```js
/**
 * @typedef {Object} Task
 * @property {string} id
 * @property {string} title
 * @property {'todo' | 'doing' | 'done' | 'archived'} status
 * @property {'low' | 'medium' | 'high'} priority
 * @property {string} createdAt
 * @property {string} updatedAt
 */
```

### Zod

Use Zod at system boundaries:

- React ↔ Go API
- React ↔ Electron preload
- extension manifest
- AI output
- import/export

Example:

```js
import { z } from 'zod'

export const TaskSchema = z.object({
  id: z.string(),
  title: z.string(),
  status: z.enum(['todo', 'doing', 'done', 'archived']),
  priority: z.enum(['low', 'medium', 'high']),
  createdAt: z.string(),
  updatedAt: z.string(),
})
```

## 12. Extension Architecture

Dao should implement internal official extensions first.

The goal is modularity, not a public plugin marketplace.

### Frontend extension interface

```js
export const extension = {
  id: 'tasks',
  name: 'Tasks',

  routes: [
    {
      path: '/tasks',
      component: TasksPage,
    },
  ],

  sidebarItems: [
    {
      label: 'Tasks',
      path: '/tasks',
      icon: 'check',
    },
  ],

  commands: [
    {
      id: 'task.create',
      title: 'Create Task',
      handler: createTask,
    },
  ],
}
```

### Go module interface

```go

	Name() string
```

## 14. AI Architecture Roadmap

AI should be built on top of existing user data.

### AI Layer 1: Summary

- note summary
- project summary
- daily summary
- weekly summary

### AI Layer 2: Retrieval

- search notes
- search project context
- retrieve activity history
- answer based on local context

### AI Layer 3: Tools

AI can call internal tools:

- createTask
- searchNotes
- getProjectStatus
- createDailyPlan

### AI Layer 4: Agent

AI can perform multi-step workflows with user confirmation.

Important rule:

```txt
AI should suggest and prepare changes first. User confirmation is required before writing major changes.
```
